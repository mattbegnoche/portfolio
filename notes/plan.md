# Software Engineering Portfolio — Astro + Tailwind + Cloudflare

## Context

The repo is currently the bare `create-astro minimal` starter (Astro 7.1.3, one placeholder `index.astro`, no styling, no content model, no deployment target configured). The goal is to build a real portfolio site showcasing software engineering projects, with:

- **Styling:** Tailwind CSS
- **Content:** Projects stored as Markdown via Astro content collections (so new projects are just new `.md` files, not code changes)
- **Pages:** Home, Projects, About, Contact
- **Contact form:** backed by a real endpoint (not just `mailto:`), deployed as a Cloudflare Worker function — the user wants to get hands-on with Cloudflare since a client is moving projects there

Confirmed against current Astro docs (Astro 7 era, mid-2026):
- Tailwind is added via `npx astro add tailwind` (Tailwind v4 + official Vite plugin — the old `@astrojs/tailwind` integration is legacy/v3-only).
- Content collections are configured in `src/content.config.ts` using `defineCollection` + `glob()` loader + a Zod schema, queried with `getCollection()` / `getEntry()` / `render()`.
- **Important, non-obvious:** the `@astrojs/cloudflare` adapter **no longer supports Cloudflare Pages** — it now targets **Cloudflare Workers** directly (deploy via Wrangler). Plan accordingly: no Pages project, no `_routes.json` dance — just `wrangler deploy`.
- Output mode stays `'static'` (the default) for the whole site; only the contact API route opts out via `export const prerender = false`. This avoids SSR-ing pages that don't need it.

## Implementation Phases

**Phase 0 — Tooling**
- `npx astro add tailwind` → Tailwind v4 via Vite plugin, create `src/styles/global.css` with `@import "tailwindcss";`
- `npx astro add cloudflare` → adds `@astrojs/cloudflare` adapter to `astro.config.mjs`, generates `wrangler.jsonc`. Keep `output: 'static'` (default, don't set to `'server'`).
- Add Vitest (`vitest`) for unit tests; add Playwright (`@playwright/test`) for the e2e smoke test.
- Add `.dev.vars` to `.gitignore`.

**Phase 1 — Base layout & design system**
- `BaseLayout.astro`: shared `<head>`, meta tags, Header/Footer slot, imports `global.css`.
- `Header.astro` (nav: Home/Projects/About/Contact), `Footer.astro` (socials/links).
- Establish Tailwind design tokens (font, color scale, spacing) directly in `global.css` via `@theme` — keep it minimal, no component library.

**Phase 2 — Content model**
- `src/content.config.ts`:
  ```ts
  const projects = defineCollection({
    loader: glob({ pattern: '**/*.md', base: './src/content/projects' }),
    schema: z.object({
      title: z.string(),
      description: z.string(),
      tags: z.array(z.string()),
      repoUrl: z.string().url().optional(),
      liveUrl: z.string().url().optional(),
      date: z.coerce.date(),
      featured: z.boolean().default(false),
      image: z.string().optional(),
    }),
  });
  export const collections = { projects };
  ```
- Add 2–3 real sample project `.md` files so pages render real content immediately.
- `ProjectCard.astro`: takes a collection entry, renders title/description/tags/links.

**Phase 3 — Pages (static content)**
- `index.astro`: hero section + `featured: true` projects pulled via `getCollection('projects', ({data}) => data.featured)`.
- `projects/index.astro`: full grid via `getCollection('projects')`, using `ProjectCard`.
- `projects/[slug].astro`: `getStaticPaths()` maps collection entries → routes; renders `entry.body` via `render(entry)`.
- `about.astro`: static bio content.
- `contact.astro`: renders `ContactForm.astro`.

**Phase 4 — Contact endpoint (the one piece of real logic)**
- `lib/validation.ts`: Zod schema `{ name, email, message }`, exported for reuse.
- `lib/email.ts`: `sendContactEmail(payload, apiKey)` — calls Resend's HTTP API via `fetch` (no Node SDK needed, keeps it Workers-compatible).
- `pages/api/contact.ts`: `export const prerender = false;` `export async function POST({ request, locals })` — parses JSON body, validates with the Zod schema (fail fast, 400 on invalid input per boundary-validation rule), reads `RESEND_API_KEY` from `locals.runtime.env` (Cloudflare adapter's env binding), calls `sendContactEmail`, returns JSON `{ success, error }` envelope.
- `ContactForm.astro`: plain HTML form + a small inline `<script>` that intercepts submit, POSTs JSON to `/api/contact`, shows success/error state — no full page reload, no framework dependency needed.
- Local secrets: `.dev.vars` holds `RESEND_API_KEY` for `astro dev`/`wrangler dev`; production secret set via `wrangler secret put RESEND_API_KEY` (a manual, account-specific step — flagged for you to run, not something I'll do automatically).

**Phase 5 — Testing**
- `validation.test.ts`: unit-tests the Zod schema directly (valid payload passes, missing/malformed fields rejected) — cheap, high-value since it's the only branching logic in the boundary layer.
- `contact-endpoint.test.ts`: integration-tests the `POST` handler with a mocked `sendContactEmail`, covering success path and validation-failure path.
- `site.spec.ts` (Playwright): one smoke test navigating Home → Projects → a project detail page → About → Contact, and submitting the contact form against a mocked/stubbed API response.
- Note: most of this site is presentational Astro markup with no branching logic — unit tests are concentrated on the contact flow rather than spread thin across static components, since that's where actual behavior (and failure modes) live.

**Phase 6 — Cloudflare deployment**
- Verify `astro build` output works against `wrangler dev` locally (exercises the real Worker runtime, not just Vite dev server).
- Document (in README) the one-time manual steps you'll need to do in the Cloudflare dashboard/CLI: `wrangler login`, `wrangler secret put RESEND_API_KEY`, and either `wrangler deploy` directly or connecting the repo via Cloudflare's git-integrated Workers Builds for auto-deploy on push — your call, both are viable and I'll leave the choice to you at that point.

## Verification

- `npm run dev` (Astro dev server) → manually click through Home/Projects/Project detail/About/Contact.
- `npx wrangler dev` → confirms the contact API route actually works against the Workers runtime (env bindings, fetch to Resend) before deploying.
- `npm run test` (Vitest) → validation + endpoint tests green.
- `npm run test:e2e` (Playwright) → smoke flow green.
- `npm run build` → production build succeeds with no type errors (`astro check`).
- Final: `wrangler deploy` → live URL, submit the real contact form once to confirm email delivery end-to-end.
